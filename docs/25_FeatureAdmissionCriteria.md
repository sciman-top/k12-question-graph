## 1. 功能准入卡

```text

是否增加教师填写负担：
是否减少教师操作步骤和选择数量：
是否需要培训才能使用：是/否
是否增加普通教师可见入口、标签、设置项或确认步骤：
```

## 2. 通过条件

高频、高价值、能明显省时间、人工负担小、培训依赖低、成本可控、可测试、可回滚，并且学生数据、版权来源、外部模型传输和备份恢复边界清晰。

教师侧流程和 UI 还必须满足：步骤更少、选择更少、术语更少、下一步明确、失败接管明确。若只是把后台复杂度换成页面上的复杂表单或说明，不算通过。

## 3. 自动后置条件

- 一年只用几次。
- 需要大量人工填写。
- 增加普通教师步骤、选择、确认或术语负担，且没有明确节省时间。
- 需要普通教师先理解系统概念、治理状态、模型配置、版本状态或数值难度区间才能完成任务。
- 新增普通教师默认入口超过四个，或把低频能力放到默认首页。
- 把脚本参数、证据 JSON、备份 manifest、迁移状态、回滚命令或权限细节直接暴露给普通教师。
- 把 `draft/test`、`candidate`、`active`、`productionEligible`、`medium_hard`、`0.55-0.7` 等内部状态、枚举或测试口径直接作为按钮、标签、提示或报告文字。
- 会引入复杂新数据结构。
- 不影响核心闭环。
- AI 成本高且结果不可靠。
- 会显著增加普通教师界面复杂度。
- 需要真实学生数据或外部模型传输才能验证，但尚未锁定合规辖区、授权和脱敏策略。

## 4. 自动不进 v0.1 条件

- 面向学生端、家长端或在线考试/监考，而不是教师 Word/Excel 工作流。
- 需要公网 SaaS、复杂部署或学校级系统对接才能体现价值。
- 需要完整标准认证、复杂 IRT、跨学科知识体系或大规模协同才能完成。
- 会让 P0/P1 无法在本机闭环验证。

## 5. 最低证据要求

功能进入开发前，至少要有：

- 对应 PRD 用户故事或任务编号。
- 教师效率理由。
- 阶段归属。
- 验收命令或手工验收样例。
- 失败接管与回滚方式。
- 数据最小化、外部传输、来源版权和备份恢复影响。
