# 11 · 教师工作流与 UX 设计

## 1. UX 最高原则

教师工作流效率最大化。界面应少操作、少学习、少配置、少打扰、多默认、多自动、多撤销、多批量、多模板。

一切流程、界面、文案和入口都必须力求简化、便捷。默认设计目标不是“展示系统全部能力”，而是让教师清楚下一步该做什么，并尽量不需要理解工程术语、脚本参数或后台治理细节。

新增或修改教师侧 UI 前，必须检查：

- 是否减少教师步骤、选择和确认次数。
- 是否隐藏或下沉了 `ImportKey`、`manifest`、`migration`、证据 JSON、备份脚本、回滚脚本等工程概念。
- 是否能由系统自动判断、给默认值、套模板或后台执行。
- 失败时是否只显示原因、影响和下一步，而不是长日志。
- 高风险动作是否仍有确认、审计和回滚，但确认文案足够简短。
- 若不能比当前流程更清楚、更省事，先重设计或后置。

## 2. 首页

普通教师默认只看到 4 个入口：

```text
导入试卷
找题组卷
导入成绩
查看分析
```

高级入口折叠：知识图谱、标签治理、模型路由、备份策略、字段管理。

普通教师首页不得因为新增能力持续扩张入口数量；新能力优先并入既有任务流，或放入高级模式/管理员入口。

## 3. 简洁模式与高级模式

| 模式 | 用户 | 功能 |
|---|---|---|
| 简洁模式 | 普通教师 | 一键导入、自动组卷、换题、导出、成绩导入、基础分析 |
| 高级模式 | 备课组长/管理员 | 细目表、知识点权重、标签治理、模型路由、备份策略 |

普通教师可见面必须只讲任务语言，不讲治理语言。以下内容只能出现在管理员、代理证据、脚本、`data-contract` 或后端合同层，不能作为普通教师按钮、标签、说明或分析页内容：

- `ImportKey`、`manifest`、`migration`、`rollback snapshot`、`active switch`。
- `candidate`、`active`、`draft_test`、`productionEligible=false`、`synthetic fixture`。
- C002R 修订治理、映射审核、知识资产健康、存储清理、缓存清理、备份回滚脚本。
- 证据 JSON 路径、脚本名、API 合同、server-state、模型路由和成本审计字段。

普通教师可见字段与状态词的硬预算见 `docs/106_TeacherVisibleMetadataBudget.md`。

普通教师看到的替代表述应是“示例数据”“示例流程”“正式启用前预览”“需确认”“可撤销”“自动检查”。UI 合同必须检查实际可见面是否外溢，而不是只检查 marker 是否存在。

## 4. 试题导入工作流

```text
上传文件
→ 系统自动处理
→ 只显示异常项
→ 教师快速确认/修正
→ 入库
```

不要让教师逐题确认全部。

## 5. 自然语言组卷工作流

```text
教师输入自然语言需求
→ AI 优化为“系统理解”
→ 显示结构化要求和细目表草稿
→ 教师确认/修改
→ 系统选题
→ 教师一键换题
→ 导出
```

不直接展示底层系统 prompt，展示“优化后的组卷要求”。

## 6. 组卷界面布局

```text
左侧：筛选栏
中间：候选题卡片
右侧：当前试卷结构/题篮
底部：预览、导出、保存
```

题目卡片显示：题型、题干前几行、题图缩略图、主知识点、难度、质量状态、最近使用、推荐理由。

## 7. 一键换题

换题保持：同知识点、同题型、相近难度、相近分值、近期未用、不与当前卷重复。所有换题可撤销。

## 8. 成绩导入工作流

```text
上传 Excel
→ 自动识别字段
→ 预览前 10 行
→ 教师确认一次
→ 保存映射模板
→ 生成分析
```

## 9. 个人默认值

TeacherPreference 自动记住：默认学科、年级、教材、组卷用途、导出模板、字体字号、Excel 映射、AI 模式、常用筛选。

## 10. 新手引导

内置示例：第一次导入试卷、第一次组卷、第一次导入成绩、第一次生成班级分析。用示例学习，不要求教师读长说明。

## 11. P0/P1 页面与状态清单

本阶段只做代理场景和工程验收，不要求真实教师现场验收。

| 页面 | P0/P1 必备状态 |
|---|---|
| 首页 | empty、loading、error、ready；普通教师只显示 4 个入口 |
| 上传页 | 文件选择、上传中、上传成功、类型不支持、文件过大、重复文件 |
| ImportJob 详情 | queued、running、succeeded、failed、retry_waiting、cancelled |
| 任务错误页 | 错误码、错误摘要、可重试/不可重试、保留原文件提示 |
| ReviewQueue | 无待处理、加载中、待确认列表、保存成功、保存失败 |
| 来源预览 | 页面加载中、页码跳转、区域高亮、截图缺失、原文件缺失 |
| 备份状态 | manifest 生成中、校验成功、校验失败、目标目录不可写 |

P0/P1 前端不做复杂视觉设计，优先保证状态明确、错误可理解、下一步动作可见。

## 12. 可访问性与课堂环境约束

教师端虽是工作台，也必须保证关键流程不被视觉细节阻断：

- 上传、任务详情、异常确认、来源预览、备份状态必须可用键盘完成核心操作。
- 错误、警告、任务进度不能只靠颜色表达，要有明确文本和状态标签。
- 表单错误要指向具体字段和下一步动作。
- 任务长时间运行时显示可刷新状态、最后更新时间和失败后继续路径。
- 默认字体、行距和点击目标要适合投影、低分辨率显示器和办公室常见缩放比例。
